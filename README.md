# Telegram бот для продажи Robux с синхронизацией с сайтом

Этот шаблон делает **бота для Telegram**, который **не дублирует бизнес-логику сайта**, а использует сайт как источник истины через общий API.

## Что уже заложено
- красивое меню Telegram-бота
- каталог пакетов Robux
- показ текущего баланса пользователя
- показ наличия Robux на складе
- создание заказа через сайт
- история заказов
- синхронизация баланса и остатков через единый backend
- каркас для привязки Telegram-аккаунта к аккаунту сайта

## Почему это правильная архитектура
Если делать покупку отдельно в боте и отдельно на сайте, логика быстро разъедется:
- на сайте один расчет цены
- в боте другой
- остатки списываются по-разному
- баланс может отличаться

Здесь же схема такая:
1. **Сайт/backend** хранит баланс, остатки Robux, заказы, платежи.
2. **Telegram-бот** только показывает интерфейс и вызывает backend API.
3. Любая покупка через сайт или Telegram сразу отражается везде.

## Что нужно на стороне сайта
Нужно, чтобы твой текущий сайт дал такие API-ручки (или очень похожие):

### 1) Привязка Telegram к аккаунту
- `POST /api/telegram/link`
- принимает `telegram_id`, `code`
- связывает пользователя сайта с Telegram

### 2) Профиль пользователя
- `GET /api/users/me?telegram_id=...`
- возвращает:
```json
{
  "id": 12,
  "username": "player",
  "balance": 1250.50,
  "currency": "RUB",
  "is_linked": true
}
```

### 3) Каталог пакетов Robux
- `GET /api/robux/packages`
- возвращает список пакетов:
```json
[
  {
    "id": 1,
    "title": "400 Robux",
    "robux_amount": 400,
    "price": 299,
    "old_price": 349,
    "in_stock": true
  }
]
```

### 4) Остаток Robux
- `GET /api/robux/stock`
- возвращает:
```json
{
  "available_robux": 152300,
  "available_packages": 41,
  "status": "ok"
}
```

### 5) Создание заказа
- `POST /api/orders/create`
- принимает:
```json
{
  "telegram_id": 123456789,
  "package_id": 1,
  "nickname": "roblox_player",
  "email": "optional@example.com"
}
```
- backend должен:
  - проверить баланс
  - проверить наличие Robux
  - создать заказ
  - списать баланс
  - уменьшить остаток Robux
  - вернуть результат

### 6) История заказов
- `GET /api/orders/my?telegram_id=...`

## Синхронизация баланса и остатков
Есть 2 нормальных варианта:

### Вариант A — лучший
**Один backend + одна база**, и сайт, и бот работают с одним и тем же API.

Плюсы:
- всё всегда синхронно
- меньше багов
- легче поддерживать

### Вариант B
Бот отдельно, но ходит в API сайта.

Плюсы:
- можно быстро запустить
- не надо ломать текущий сайт

Минус:
- нужно аккуратно защитить API

## Как связать Telegram с сайтом
Лучший UX:
1. Пользователь заходит на сайт.
2. В личном кабинете нажимает **Привязать Telegram**.
3. Сайт генерирует одноразовый код, например `RBX-483912`.
4. Пользователь отправляет код боту.
5. Бот вызывает `POST /api/telegram/link`.
6. После этого баланс и заказы пользователя доступны и на сайте, и в боте.

## Запуск
```bash
cp .env.example .env
pip install -r requirements.txt
python -m app.main
```

## Что надо адаптировать под твой проект
1. Подставить реальный URL твоего backend.
2. Подстроить JSON-ответы под твой сайт.
3. Подключить реальную авторизацию API.
4. Доделать оплату/пополнение баланса, если у тебя это уже есть на сайте.

## Что я бы сделал в твоем проекте прямо сейчас
- вынес бы покупку Robux в единый сервис `robux_service`
- сайт и бот дергали бы именно его
- баланс, остатки и заказы держал бы в одной БД
- Telegram сделал бы как второй интерфейс продаж, а не как отдельную систему


## Тестовая привязка по ID

Для быстрого теста без кода привязки можно включить переменную `TEST_SITE_USER_ID`.

Пример:

```env
TEST_SITE_USER_ID=1
```

Тогда кнопка «Привязать аккаунт» не будет спрашивать код, а бот начнет отправлять в backend оба параметра: `telegram_id` и `site_user_id=1`.

Чтобы это заработало, backend должен уметь принимать `site_user_id` как приоритетный идентификатор в тестовом режиме.
